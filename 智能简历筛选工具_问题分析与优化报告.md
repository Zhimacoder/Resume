# 智能简历筛选工具 — 问题分析与优化报告

**文档版本**：1.0  
**分析日期**：2026-06-15  
**分析范围**：V1.1 PRD 需求文档 + 实际代码实现（backend/frontend/config）+ 运行日志（logs/app.log）

---

## 一、总体评估

工具整体架构清晰，前后端分离合理，核心业务流程（上传 → OCR → 脱敏 → 大模型分析 → 结果展示）可以跑通。API Key 全局状态管控、数据保活缓存等 V1.1 核心需求均有落地实现。但在**安全性、健壮性、需求一致性**三个维度上存在多处问题，部分问题会直接影响功能可用性或数据安全。

---

## 二、已确认问题清单

### 2.1 安全性问题（高优先级）

**问题 1：加密密钥明文存储，等同于未加密**

- **位置**：`config/config.py` → `_get_cipher()` 函数；`config/.key` 文件
- **描述**：加密密钥 `.key` 文件与加密后的配置文件 `models.json.enc` 存放在同一目录下，攻击者拿到项目目录即可同时拿到密钥和密文，Fernet 加密形同虚设。API Key 实际处于裸露状态。
- **风险等级**：高

**问题 2：`.key` 文件未加入 `.gitignore`**

- **位置**：`.gitignore` 文件；`config/.key`
- **描述**：当前 `.gitignore` 未明确排除 `config/.key` 和 `config/models.json.enc`，一旦提交到 Git 仓库，API Key 将永久泄露（即使后续删除，历史记录中仍可找回）。
- **风险等级**：高

**问题 3：CORS 配置过于宽松**

- **位置**：`backend/main.py` → `CORSMiddleware`，`allow_origins=["*"]`
- **描述**：允许任意来源跨域请求，在本地工具场景下，其他恶意网站可向后端接口发起请求，存在 CSRF 风险。
- **风险等级**：中

---

### 2.2 功能缺陷（与 PRD 不符）

**问题 4：Export Excel 功能运行报 500 错误（日志实证）**

- **位置**：`backend/main.py` → `/api/export` 接口；`logs/app.log`
- **描述**：日志中明确记录了 `POST /api/export HTTP/1.1 500 Internal Server Error`。接口内部以 `try/except ImportError` 包裹 openpyxl 导入，推测是生产环境未安装 openpyxl，但错误信息对用户不友好，且 `requirements.txt` 未将 openpyxl 列为强依赖（仅列 `PyMuPDF`、`python-docx` 等），导致安装后缺包。
- **风险等级**：高（功能直接不可用）

**问题 5：百度文心 API Key 字段语义错误**

- **位置**：`backend/llm.py` → `WenxinCaller.call()`
- **描述**：文心接口使用 AK/SK 换取 access_token，但代码将 `self.api_key` 同时传给 `client_id` 和 `client_secret`，即用一个字段存两个值，导致 token 获取必然失败。PRD 的配置页仅提供单一 API Key 字段，并未针对文心做 AK/SK 双字段的说明。
- **风险等级**：高（百度文心完全无法使用）

**问题 6：PRD 要求简历格式为 doc/docx/pdf，实现支持更多格式但 PRD 未覆盖**

- **位置**：PRD 2.2.1 vs `frontend/index.html` 文件 accept 属性、`backend/parser.py`
- **描述**：实现端支持 txt、rtf、jpg、png 等图片格式，但 PRD 中仅提及 doc/docx/pdf。虽然额外支持是好事，但 PRD 与实现不一致，后续迭代容易混乱，且 PRD 中的「后续迭代方向」里还写着「支持更多格式如 txt/rtf/图片」（说明本版本不应包含）。

**问题 7：结果页「返回主页」不清空历史结果数据**

- **位置**：`frontend/result.html` → `goHome()` 函数
- **描述**：PRD 2.3.2 明确「点击返回主页，不保留上一次筛选的临时数据」，但 `goHome()` 仅执行 `window.location.href = '/'`，并未调用 `sessionStorage.removeItem('screening_results')`，旧的筛选结果依然驻留在 sessionStorage，直接进入结果页仍可看到旧数据。
- **风险等级**：中

**问题 8：文件缓存在大文件场景下会导致 sessionStorage 溢出**

- **位置**：`frontend/js/cache.js` → `fileToCacheable()`；`frontend/index.html` → `saveToCache()`
- **描述**：将简历文件内容以 base64 编码存入 sessionStorage，10 份 10MB 的简历将产生约 137MB 数据，而浏览器 sessionStorage 上限通常为 5–10MB，溢出后静默失败（代码只 `console.error` 不提示用户），导致缓存机制完全失效。
- **风险等级**：中

**问题 9：批量处理为串行，超时风险大**

- **位置**：`backend/main.py` → `/api/screening` 接口
- **描述**：10 份简历在 `for file in files` 循环内串行调用大模型，每次调用 timeout=60s，极端情况下总耗时可达 600s，远超 PRD 要求的「10 份总处理时间 ≤ 2 分钟」。且接口为同步阻塞，FastAPI 的 async 特性完全未被利用。
- **风险等级**：高

### 2.3 代码健壮性问题

**问题 10：大模型响应解析策略脆弱**

- **位置**：`backend/llm.py` → `LLMCaller._parse_response()`
- **描述**：使用 `re.search(r'\{[\s\S]*\}', response_text)` 提取 JSON，该正则匹配第一个 `{` 到最后一个 `}`，若模型在 JSON 前后输出了多余文本且文本中包含花括号（如代码示例），将提取到错误内容，`json.loads` 解析失败。应改用专门提取 markdown 代码块的正则或使用 `response_format={"type":"json_object"}`（支持该参数的模型）。

**问题 11：全局状态校验双重冗余**

- **位置**：`frontend/js/state.js` → `isValid()` 函数
- **描述**：`isValid()` 内部先判断 `state.is_config_valid`，再取 `model.is_valid` 和字段非空做二次判断，逻辑正确但冗余，若后端 `save_config` 的 `is_config_valid` 计算逻辑与前端 `isValid()` 不一致，将出现前后端状态不同步的 bug。建议统一以后端 `/api/config/status` 的 `is_config_valid` 为唯一真相来源。

**问题 12：doc 格式文件兼容性差**

- **位置**：`backend/parser.py` → `parse_doc()`；`requirements.txt`
- **描述**：依赖 `antiword` 库解析 `.doc` 文件，但 antiword 在 Windows 环境下安装困难，macOS 需 homebrew，且 requirements.txt 未包含该依赖。代码中的降级逻辑（先尝试 docx 解析）在真实 `.doc` 文件上通常不会成功，实际上 `.doc` 在大多数环境中无法解析。

**问题 13：OCR 仅支持本地 Tesseract，中文识别效果差**

- **位置**：`backend/ocr.py`
- **描述**：仅实现了 pytesseract 本地 OCR，PRD 风险项中提到「优先使用第三方 OCR API，本地 OCR 作为备选」，但实现中并无第三方 OCR 路径，且 Tesseract 对中文简历（特别是图片格式）的识别准确率偏低。

**问题 14：requirements.txt 与实际依赖不一致**

- **位置**：`backend/requirements.txt`
- **描述**：openpyxl、antiword、striprtf 均未列入 requirements.txt，但代码中实际使用（导出 Excel 需 openpyxl，doc 解析需 antiword，rtf 解析需 striprtf）。新用户按 requirements.txt 安装后，上述功能全部报错。

**问题 15：start.sh 端口冲突处理缺失**

- **位置**：`start.sh`；`logs/app.log`
- **描述**：日志记录了 `[Errno 48] error while attempting to bind on address ('0.0.0.0', 8000): address already in use` 错误，`start.sh` 未加入端口占用检测与自动 kill 逻辑，重复启动时服务无法拉起，用户体验差。

---

### 2.4 用户体验问题

**问题 16：加载状态不显示逐个进度**

- **位置**：`frontend/index.html` → `handleScreening()`
- **描述**：筛选 10 份简历时，loading 文案固定为「正在分析简历，请稍候...」，用户无法感知当前处理到第几份，等待期间容易误以为页面卡死。

**问题 17：文件重名校验仅对比文件名，不对比内容**

- **位置**：`frontend/index.html` → `handleFiles()`
- **描述**：`uploadedFiles.some(f => f.name === file.name)` 仅靠文件名去重，用户从不同路径上传同名不同内容的简历会被错误拦截。

**问题 18：无 favicon.ico，控制台报 404**

- **位置**：`logs/app.log`：`GET /favicon.ico HTTP/1.1 404 Not Found`
- **描述**：缺少 favicon，每次页面访问均产生 404 请求，污染日志，对品牌形象也有轻微影响。

## 四、优化方向与内容

### 4.1 架构层面

**异步批处理重构**

当前同步串行处理简历是性能瓶颈的根本原因。建议将 `/api/screening` 接口改造为任务队列模式：提交请求后立即返回 `task_id`，前端轮询 `/api/task/{task_id}` 获取进度，后端使用 `asyncio.gather` 或 `concurrent.futures.ThreadPoolExecutor` 并发处理各份简历。这一改造同时可解决超时、进度无反馈两个问题。

**配置存储升级**

当前用 Fernet 加密 JSON 文件的方案安全性存在密钥保管漏洞。长期建议改用操作系统密钥链（Python `keyring` 库，跨平台支持 macOS Keychain / Windows Credential Manager / Linux Secret Service），API Key 永不落盘。

### 4.2 功能层面

**APIKey 连通性测试增强**

PRD 8 后续迭代中提及，但测试接口（`/api/test-connection`）实际已实现，Config 页面也有调用。可考虑提前开放给用户（去掉「暂不考虑」限制），并在配置页为每个模型显示连通测试状态标识（✅/❌），显著提升配置效率。

**脱敏策略增强**

当前 `desensitizer.py` 的姓名识别仅匹配固定格式（「姓名：XX」），无法覆盖简历开头直接写姓名的常见格式。建议结合 NLP 命名实体识别（如 `jieba` 词性标注或轻量级 NER 模型）提升姓名识别召回率，同时增加微信号、QQ 号的脱敏规则。

**结果对比视图**

结果页目前只能逐个查看，当候选人数量较多时，HR 希望横向对比多份简历的核心指标（分数、匹配点）。可在结果页增加「对比模式」，支持同时选中 2–4 份简历并排展示关键维度。

**匹配度校准**

大模型评分存在随机性，同一份简历多次评分可能差距较大。可对每份简历调用模型 2–3 次取平均分（成本换稳定性），或在 Prompt 中加入「请按以下评分标准严格打分」的量化锚点，减少波动。

### 4.3 工程质量层面

**统一错误处理**

各 LLMCaller 子类的异常处理均为 `print` + `return None`，前端收到 `null` 结果时只能提示「大模型分析失败」，无法区分是网络超时、API Key 错误还是额度耗尽。建议改为结构化异常（自定义 `LLMError` 类，携带 `error_code` 和 `message`），后端透传给前端，便于用户自助排查。

**日志规范化**

目前后端日志混杂 `print` 和 uvicorn 的 access log，没有统一格式，调试困难。应全面改用 Python `logging` 模块，区分 DEBUG/INFO/WARNING/ERROR 级别，并配置日志轮转（`RotatingFileHandler`）防止日志文件无限增长。

**前端代码组织**

`index.html` 将所有 JS 业务逻辑内联在 `<script>` 中，共约 300 行，与 `state.js`、`cache.js`、`api.js` 三个模块分离的风格不一致。建议将内联 JS 抽取为 `frontend/js/index.js`，保持统一的模块化风格，便于后续维护。

**单元测试缺失**

目前项目无任何测试文件。建议优先为以下模块补充单测：`desensitizer.py`（脱敏规则正确性）、`parser.py`（各格式解析）、`llm.py` 的 `_parse_response()`（JSON 提取健壮性）。可使用 `pytest` + `pytest-asyncio`，结合 mock 隔离外部 API 依赖。
