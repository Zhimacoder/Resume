## 1. 项目概述

### 1.1 项目背景
企业招聘过程中，HR需人工筛选大量简历，效率低且易受主观因素影响。为提升简历筛选效率，降低人工成本，开发基于大模型的智能简历筛选工具。

### 1.2 产品目标
- 支持批量上传简历（≤10份），自动完成OCR识别与信息脱敏
- 基于大模型分析简历与JD的匹配度，输出结构化筛选结果
- 提供APIKey配置页面，支持主流大模型灵活切换
- **新增**：APIkey配置状态全流程管控，未配置时强提醒+操作拦截，已配置时无干扰，保障用户操作数据不丢失

### 1.3 目标用户
企业HR、招聘专员及需要快速筛选简历的业务人员

## 2. 功能模块

### 2.1 模块总览
| 模块 | 功能描述 |
|------|----------|
| 主页（页面1） | JD填写、简历上传、智能筛选触发、APIkey状态醒目提醒、操作拦截 |
| 结果页（页面2） | 简历列表展示、详情查看、匹配度分析 |
| API配置页（页面3） | 主流大模型APIKey配置、自定义模型配置、配置保存与状态同步 |
| 全局状态管控 | APIkey配置状态校验、页面数据保活、拦截弹框管理 |

### 2.2 页面1 - 主页功能规范

#### 2.2.1 JD填写
- 文本输入框，支持粘贴/手动输入岗位JD，无字数限制
- 占页面宽度80%

#### 2.2.2 简历上传
- 支持格式：doc、docx、pdf
- 数量限制：一次最多上传10份
- 交互：支持拖拽上传或点击按钮选择文件
- 上传后显示文件名列表

#### 2.2.3 智能筛选按钮
- 点击后先执行APIkey状态前置校验
- 校验通过后触发：OCR识别 → 信息脱敏 → 调用大模型 → 跳转结果页

#### 2.2.4 APIkey状态醒目提醒模块
- **展示位置**：页面顶部通栏（标题下方）
- **展示规则**：仅未配置状态下渲染展示，已配置状态下完全隐藏
- **视觉规范**：红色渐变背景、白色加粗字体、内边距12px、通栏100%宽度
- **内容**：【⚠️ 请先完成APIkey配置，否则无法使用智能筛选功能】
- **右侧按钮**：【去配置APIkey】，点击后跳转至API配置页

#### 2.2.5 未配置状态操作拦截弹框
- **触发条件**：未配置APIkey时，用户点击【智能筛选】按钮
- **弹框层级**：页面最高层级，居中展示，遮罩覆盖全页面
- **标题**：缺少APIkey配置
- **正文**：当前未完成大模型APIkey有效配置，无法执行智能筛选，请先完成配置后再操作。
- **按钮组**：
  1. 【取消】：关闭弹框，完整保留用户已填写的JD内容、已上传的简历文件
  2. 【去配置APIkey】：主按钮，点击后跳转至API配置页

#### 2.2.6 页面数据保活机制
- **缓存载体**：sessionStorage
- **保活数据**：JD文本内容、简历文件列表、文件解析临时缓存
- **保活生效场景**：
  1. 跳转至API配置页时自动触发数据缓存
  2. 从API配置页返回页面1时自动触发数据回填
- **缓存有效期**：当前浏览器会话全程有效
- **清空时机**：用户主动点击【清空内容】、完成智能筛选全流程、手动刷新页面

#### 2.2.7 页面布局
- 顶部：标题"智能简历筛选工具" + 右上角【API配置】固定入口按钮
- 警示通栏：APIkey未配置时展示，已配置时不渲染
- 中部：JD输入框（占页面宽度80%）
- 下部：简历上传区域（拖拽区+文件列表）+ "智能筛选"按钮

### 2.3 页面2 - 简历筛选结果页

#### 2.3.1 左侧布局（30%宽度���
- 展示简历名称列表，按匹配度从高到低排序
- 点击简历名称，右侧切换至对应详情

#### 2.3.2 右侧布局（70%宽度）
展示当前选中简历的详细信息：
1. 简历名称
2. 简历概述（大模型生成的简历核心信息摘要）
3. 与岗位的匹配点（分点列出）
4. 与岗位的不足点（分点列出）
5. 最适合的3个面试问题（分点列出）

#### 2.3.3 交互逻辑
- 页面加载时默认展示排名第一的简历详情
- 左侧列表支持滚动
- 右侧详情支持滚动
- 页面左上角提供【返回主页】按钮

### 2.4 页面3 - APIKey配置页

#### 2.4.1 主流大模型配置
支持以下模型的APIKey输入：
- 百度文心、阿里千问、字节豆包、智谱GLM、MinMax
- 每个模型对应独立输入框，输入后保存至本地

#### 2.4.2 自定义模型配置
支持用户输入：
- 模型名称、API Endpoint、APIKey
- 保存后可在筛选时选择自定义模型

#### 2.4.3 模型选择
- 提供下拉框，选择当前用于筛选的大模型
- 默认选中第一个配置完成的有效模型

#### 2.4.4 配置保存与校验
- 点击【保存配置】按钮时，先执行字段校验
- APIKey、Endpoint为必填项，非空才允许保存
- 保存成功后，实时更新全局APIkey配置状态
- 保存失败时，弹出错误提示

#### 2.4.5 页面返回入口
- 页面顶部固定【返回主页】按钮
- 点击后直接跳转回页面1，触发页面1的数据回填与状态校验

## 3. 全局状态管理规范

### 3.1 状态定义
| 状态 | 判定规则 |
|------|----------|
| 未配置状态 | 无任何大模型配置信息，或当前选中的默认模型无有效、非空的APIkey+Endpoint完整配置 |
| 已配置状态 | 当前选中的大模型已完成APIkey+Endpoint配置，且字段非空、格式校验通过 |

### 3.2 全局校验规则
- 校验触发时机：页面初始化加载、用户触发核心操作、路由页面跳转、配置信息保存/修改后
- 校验优先级：全局状态校验优先于所有业务操作执行
- 状态同步：配置信息修改后，实时同步全局状态

## 4. 技术架构

### 4.1 技术栈
| 层级 | 技术选型 |
|------|----------|
| 前端 | HTML + CSS + JavaScript（原生），sessionStorage/localStorage做数据缓存与状态管理 |
| 后端 | Python 3.8+，FastAPI |
| OCR识别 | PyTesseract（本地OCR）/ 百度OCR API |
| 文档解析 | PyPDF2（PDF）、python-docx（docx）、antiword（doc） |
| 大模型调用 | Requests（HTTP请求调用大模型API） |
| 配置存储 | 后端本地加密JSON配置文件，前端同步配置状态缓存 |

## 6. 非功能需求

### 6.1 性能需求
- 单份简历OCR + 大模型响应时间 ≤ 10秒
- 批量上传10份简历时，总处理时间 ≤ 2分钟
- 页面跳转与数据回填耗时 ≤ 300ms

### 6.2 兼容性需求
- 前端：支持Chrome、Firefox、Edge最新版本浏览器
- 后端：Python 3.8+，兼容Windows/Linux/macOS

### 6.3 安全性需求
- APIKey加密存储至本地配置文件
- 简历数据仅在本地处理，仅传输脱敏后文本至大模型
- 本地缓存数据仅在当前会话有效

### 6.4 可用性需求
- 页面操作流程 ≤ 3步
- 提供操作提示（如"上传中""分析中"等加载状态）
- 配置往返操作无数据丢失
